# OpenAPI / Swagger Reference

The OmniExtract Gateway API is fully documented via Swagger UI and ReDoc, automatically generated by FastAPI.

## Interactive Documentation

| Interface | URL |
| :--- | :--- |
| Swagger UI | `http://localhost:8000/docs` |
| ReDoc | `http://localhost:8000/redoc` |
| OpenAPI JSON | `http://localhost:8000/openapi.json` |

## Exporting the OpenAPI Schema

```bash
# Export to JSON
curl http://localhost:8000/openapi.json > docs/openapi.json

# Export to YAML (requires yq)
curl http://localhost:8000/openapi.json | yq -P > docs/openapi.yaml
```

## Key Endpoints

### `POST /extract`

Route a document extraction request to any supported engine.

**Request body:** `ExtractionRequest`  
**Response:** `StandardizedExtraction`

### `GET /prebuilts`

List all available prebuilt document templates.

### `POST /prebuilts/custom`

Register a new custom prebuilt template. The gateway auto-generates the optimal LLM extraction prompt from the provided field list.

### `GET /registry/models`

Return the full model registry grouped by provider.

### `GET /health`

Health check endpoint.

## Ejemplo LLM (`provider=llm_router`)

### Request

```json
{
  "document": "<base64_o_url>",
  "engine_config": {
    "provider": "llm_router",
    "model": "gpt-4o",
    "api_keys": {
      "api_key": "<SECRET>"
    }
  },
  "extraction_target": {
    "document_type": "invoice",
    "custom_fields": ["invoice_number", "total_amount"]
  }
}
```

### Response (estándar)

```json
{
  "raw_text": "Invoice #123 ...",
  "fields": [
    {
      "key": "invoice_number",
      "value": "123",
      "confidence": 0.98,
      "bounding_box": {
        "x0": 0.1,
        "y0": 0.2,
        "x1": 0.3,
        "y1": 0.25
      }
    }
  ],
  "tables": [],
  "engine_used": "llm_router:gpt-4o",
  "processing_time_ms": 142.55
}
```


### Notas operativas del adapter LLM

- `custom_fields` (si se envía) tiene prioridad sobre `required_fields` del prebuilt.
- El adapter filtra claves reservadas en `api_keys` para evitar sobreescritura de `model/messages/timeout/max_tokens`.
- El request a LiteLLM fuerza `response_format={"type":"json_object"}` para mejorar consistencia.
